4. ROOT CAUSE ANALYSIS
================================================================================

  RC-1: ANTHROPIC API KEY BROKEN — ALL AGENTS ON GEMINI FALLBACK
  ──────────────────────────────────────────────────────────────
  PM2 logs show repeated:
    "All models in fallback chain for claude-opus-4-1 are down — using preferred anyway"

  The Anthropic API key was quoted in .env (fixed during setup), but the
  bot still couldn't authenticate. Every agent fell back to Gemini Flash
  or Vertex Gemini. Vertex Gemini then returned HTTP 401 (OAuth2 credential
  failure), leaving agents with the worst-quality fallback.

  Impact: Agents on Gemini Flash produce shorter, less structured responses
  that miss the pattern matchers. Opus-tier agents (Ace, Jude, Mia, Leo)
  are most affected because their tests assume Opus-quality output.

  Fix: Verify ANTHROPIC_API_KEY is actually valid (not just unquoted).
  Regenerate at console.anthropic.com if needed. Also fix Vertex Gemini
  OAuth — the service account may need `roles/aiplatform.user` on the
  GCP project, or the ADC token may have expired on the VM.


  RC-2: AGENT CHANNEL TIMEOUTS (PHASE 2 DEVASTATION)
  ──────────────────────────────────────────────────
  In Phase 2, tests are sent directly to agent channels (e.g., 💻-developer).
  The text channel handler has a 120-second timeout (TEXT_AGENT_RESPONSE_TIMEOUT_MS).
  With Gemini fallback, many agents:
    1. Start with "Thinking…"
    2. Hit the 120s agent timeout before producing a response
    3. Tester then waits for the test-level timeout (180–300s)
    4. No further messages appear → "idle timeout" or "hard ceiling reached"

  This explains why Phase 2 had 43% pass rate while Phase 1 (groupchat
  with Riley) had 84%. Riley routes + responds quickly; specialist agents
  on their own channels with degraded models are too slow.

  Fix: Increase TEXT_AGENT_RESPONSE_TIMEOUT_MS to 300s for Opus agents.
  Implement per-agent timeout scaling based on model quality. If agent
  detects it's on fallback model, auto-increase timeout.


  RC-3: TOOL AUDIT CALLBACK NOT FIRING FOR SOME TOOLS
  ───────────────────────────────────────────────────
  9 tests failed with TOOL_AUDIT_MISSING. The tool audit callback posts
  [TOOL:toolname] to #terminal, but some tools never triggered it:
    - memory_read, memory_append (Riley)
    - send_channel_message (Riley)
    - edit_file, read_file (Riley self-edit test)
    - git_create_branch (Riley create-branch-pr test)
    - search_files (Leo android-build-review)
    - run_command (Kane dependency-scan)
    - fetch_url (Sophie app-ux-audit)
    - mobile_harness_* (Max mobile tests)

  Two possible causes:
    a) Agent decided not to use the tool (prompt wasn't specific enough)
    b) Tool call errored before the audit callback fired

  Fix: Review test prompts to be more directive ("Use the memory_read
  tool to…"). Add audit logging at tool entry, not just on success.


  RC-4: UPGRADES-POST TOKEN ECHO FAILURE
  ──────────────────────────────────────
  Three agents (Ace, Sophie, Jude) failed upgrades-post. The test sends
  a prompt containing a smoke token (e.g., SMOKE_DEVELO_UPGRA_123456) and
  expects the agent to include that token in its #upgrades post.

  With Gemini fallback, agents either:
    a) Don't post to #upgrades at all (confused about the task)
    b) Post but reformulate the message, dropping the smoke token

  Fix: Make the test prompt more explicit: "Post the following exact text
  to #upgrades including this token: SMOKE_XXX". Alternatively, match on
  semantic content rather than exact token echo.


  RC-5: PATTERN MATCHERS TOO STRICT FOR FALLBACK MODEL OUTPUT
  ───────────────────────────────────────────────────────────
  51 PATTERN_MISMATCH failures. Many tests use expectAll/expectAny with
  regex patterns that assume Opus-quality structured output. When agents
  use Gemini Flash, they produce:
    - Shorter responses
    - Less structured format (no "Result:" / "Evidence:" headers)
    - Tool results embedded differently
    - More conversational tone

  These valid (if lower-quality) responses don't match the regex.

  Fix: Broaden regex patterns to accept both Opus and Flash formatting.
  For critical tests, add fallback patterns. Consider a "model-aware"
  matcher that relaxes expectations when agents are on fallback models.


  RC-6: GOAL MODE — ONLY 2/13 AGENTS PARTICIPATED
  ───────────────────────────────────────────────
  The freeform goal phase (build Design System page) ran for 12 minutes.
  Only Riley and Ace responded (21 messages total). The other 11 agents
  never engaged because:
    1. Riley delegated exclusively to Ace (developer task)
    2. No explicit fan-out to specialists like Sophie (UX), Max (QA),
       Kane (security), Kai (performance)
    3. The goal prompt said "coordinate all 13 agents" but Riley
       optimised by sending work to the most relevant agent only

  This is actually good delegation behaviour. But if the intent is to
  test all agents, the goal prompt needs explicit per-agent tasks.

  6.1  Model fallback — Increase text channel timeout for Opus agents
  ────────────────────────────────────────────────────────────────────
  File: src/discord/handlers/textChannel.ts

  When an agent's preferred model is unavailable and it falls back to a
  slower model, the 120-second timeout is too aggressive. Add:

    const timeoutMs = isOnFallbackModel(agentId)
      ? TEXT_AGENT_RESPONSE_TIMEOUT_MS * 2
      : TEXT_AGENT_RESPONSE_TIMEOUT_MS;

  Or make TEXT_AGENT_RESPONSE_TIMEOUT_MS per-agent configurable:
    Opus agents: 300s

  6.2  Tool audit — Log at tool entry, not only on success
  ────────────────────────────────────────────────────────
  File: src/discord/bot.ts (setToolAuditCallback)

  Currently the audit callback may not fire if the tool throws before
  completion. Move the [TOOL:toolname] post to the beginning of tool
  execution rather than the end. Add a second post on completion with
  the result status.


  6.3  Memory tools — Riley's tool invocation
  ──────────────────────────────────────────
  File: src/discord/tools.ts

  Riley has memory_read and memory_append in her toolset, but she's not

  Add to Riley's agent prompt: "When asked to read or write memory, use

  6.4  send_channel_message — Verify channel resolution
  ─────────────────────────────────────────────────────
  File: src/discord/tools.ts

  The send_channel_message tool resolves channels by name. If channel
  names have emojis (e.g., "📈-upgrades"), the lookup may fail silently.

7. TEST FRAMEWORK CHANGES REQUIRED
================================================================================

  7.1  Broaden PATTERN_MISMATCH regex patterns
  ────────────────────────────────────────────
  51 tests failed due to strict regex. Recommendations:

  a) evidence-format-contract: Change /\bresults?\s*:/i to also accept
     "## Result", "**Result**", "Outcome:", "Finding:" etc.

  b) Specialist tests (postgres-safety, ios-stack, etc.): Instead of
     requiring specific keywords, check for semantic markers:
       Old: /\bWAL\b.*\breplication\b/i
       New: /\b(WAL|write.ahead|replication|standby|backup)\b/i

  c) GCP tool tests: Many fail because the agent discusses the tool
     result conversationally instead of showing raw output. Accept both.


  7.2  Add model-awareness to test expectations
  ─────────────────────────────────────────────
  Add a field to test definitions:
    modelFloor?: 'opus' | 'flash' | 'any'

  If the agent is known to be on a fallback model, skip tests that
  require Opus-level output or use relaxed patterns:
    if (currentModel === 'gemini-2.0-flash' && test.modelFloor === 'opus') {
      return { ok: true, reason: 'skipped (model below floor)' };
    }


  7.3  Fix idle timeout vs hard ceiling classification
  ───────────────────────────────────────────────────
  The current system classifies both timeout types as PATTERN_MISMATCH
  (since the response existed but lacked patterns). These should be
  classified as TIMEOUT so suggestFix() gives timeout-specific advice.

  Check: if snippet contains "idle timeout" or "hard ceiling reached",
  categorize as TIMEOUT, not PATTERN_MISMATCH.


  7.4  Extend Phase 2 timeouts for specialist agents
  ──────────────────────────────────────────────────
  Phase 2 tests use 180s default timeout. Specialist agents on Flash
  models may need 300s. Set per-agent-role timeout overrides:
    if (agentRole === 'review') timeoutMs = 240_000;
    if (agentRole === 'full')   timeoutMs = 360_000;


  7.5  Fix the rerun-failed stage not executing
  ────────────────────────────────────────────
  File: scripts/overnight-smoke.sh

  The overnight script exits after PM2 log capture without running
  rerun-failed. Likely the --pm2-logs flag causes the node process to
  exit before the rerun stage starts. Fix: separate PM2 capture from
  the rerun stage:

    # Stage 2: Matrix
    node dist/discord/tester.js --profile=full-matrix
    # Stage 2b: Capture PM2 logs (separate invocation)
    node dist/discord/tester.js --pm2-logs
    # Stage 3: Rerun failed
    node dist/discord/tester.js --rerun-failed


  7.6  Upgrades-post test — Use semantic matching, not token echo
  ──────────────────────────────────────────────────────────────
  Instead of requiring agents to echo an exact smoke token, check that
  the agent called send_channel_message with target='upgrades'. This
  is already available via tool audit. Change:
    Old: expectUpgradesPost with token
    New: expectToolAudit: ['send_channel_message'] + check tool args
         contain 'upgrades'


  7.7  TOOL_AUDIT_MISSING — Make prompts more directive
  ────────────────────────────────────────────────────
  For tool audit tests, the prompt must explicitly require tool use:
    Bad:  "Read the repository memory"
    Good: "Use the memory_read tool to read the repo memory index"

  Update prompts for all 9 failing TOOL_AUDIT_MISSING tests to name
  the specific tool expected.

8. DISCORD UX IMPROVEMENTS
================================================================================

  8.1  RESPONSE FORMATTING — Structured embeds instead of plain text
  ─────────────────────────────────────────────────────────────────
  Current state: Agents send plain text via webhooks (up to 2000 chars).
  Long responses get clipped with "…". No visual hierarchy.

  Recommendation:
    • Use Discord embeds for structured responses (title, description,
      fields, footer, colour coding by agent)
    • Code blocks → use ```lang syntax highlighting
    • Tool results → collapsible details in embed fields
    • Error states → red-coloured embed with error title
    • Success states → green embed with checkmark

  Implementation:
    In textChannel.ts, convert agent responses to EmbedBuilder format:
      const embed = new EmbedBuilder()
        .setColor(agent.color)
        .setAuthor({ name: agent.name, iconURL: agent.avatarUrl })
        .setDescription(response.slice(0, 4096))
        .setFooter({ text: `Model: ${modelUsed} | ${tokenCount} tokens` });


  8.2  PROGRESS INDICATORS — Replace "Thinking…" with rich status
  ───────────────────────────────────────────────────────────────
  Current state: Agent posts "Thinking…" → then edits to "⏳ still
  working…" after 15s → then final response. Looks static and uncertain.

  Recommendation:
    • Phase 1: "🔍 Reading codebase…" (when tool calls are executing)
    • Phase 2: "💭 Analysing results…" (when LLM is generating)
    • Phase 3: "✍️ Composing response…" (when formatting output)
    • Timeout: "⏱️ This is taking longer than expected — still working"
    • Final timeout: "❌ Response timed out — please try again"

  Expose tool execution state in the progress message so users know
  the agent is doing real work, not stalled.


  8.3  ERROR HANDLING — Clear failure states
  ─────────────────────────────────────────
  Current state: When an agent errors, the "Thinking…" message may stay
  forever, or the error is logged to console but hidden from the user.

  Recommendation:
    • On timeout: Edit "Thinking…" to a red embed explaining the timeout
    • On model error: Show which model failed and which fallback is active
    • On tool error: Show which tool failed and the error category
    • Never leave a "Thinking…" message unresolved — always clean up


  8.4  THREAD MANAGEMENT — Better thread naming and lifecycle
  ──────────────────────────────────────────────────────────
  The thread-name-quality and thread-auto-close tests passed, which is
  good. To improve further:
    • Include agent name in thread title: "Ace: Fix login bug"
    • Auto-archive threads after 24h of inactivity
    • Pin summary message at thread start
    • Add a "📌 Thread Summary" embed when thread closes


  8.5  BUTTON INTERACTIONS — Decision trees and confirmations
  ──────────────────────────────────────────────────────────
  The decision-buttons-render test passed. Improvements:
    • Add "Are you sure?" confirmation before destructive actions
    • Disable buttons after they've been clicked (prevent double-click)
    • Show button click feedback immediately (edit message to show
      "Processing…" with the selected option highlighted)
    • Add timeout on button interactions (disable after 5 minutes)


  8.6  CHANNEL HYGIENE — Keep channels clean and informative
  ────────────────────────────────────────────────────────
  The channel_hygiene check failed because #upgrades had 0 messages.

  Recommendations:
    • Post a daily summary to each agent channel: "📊 Today's activity:
      X tasks completed, Y in progress, Z blocked"
    • #upgrades should have a pinned message explaining its purpose
    • Auto-archive inactive channels with a "This channel has been
      quiet — resuming when new work arrives" message
    • #terminal should have message grouping/batching to avoid noise
      (currently 429 ASAP system events flood the channel)


  8.7  MULTI-AGENT VISIBILITY — Show delegation chain
  ──────────────────────────────────────────────────
  When Riley delegates to Ace, users in #groupchat see Riley's message
  but may not know Ace is working in another channel. Suggest:
    • Post a delegation notice: "🔄 Delegating to Ace in 💻-developer"
    • When Ace finishes, post result back to #groupchat with attribution
    • Show a "task pipeline" embed: Riley → Ace → Max → Done
    • Add reaction-based status: 🔄 in progress, ✅ complete, ❌ failed


  8.8  MESSAGE LENGTH — Smart truncation and pagination
  ───────────────────────────────────────────────────
  Discord has a 2000-char message limit and 4096-char embed limit.
  Current behaviour clips at 1900 chars with "…".

  Recommendation:
    • Split long responses into multiple messages with "1/3", "2/3" labels
    • Use a "📄 Full response" button that sends a DM with the complete text
    • For code output, use a Gist/pastebin link for >50 lines
    • For tool results (e.g., ls output), use collapsible embed fields


  8.9  AGENT IDENTITY — Visual consistency and personality
  ──────────────────────────────────────────────────────
  The webhook format (emoji + name + avatar) works well. Enhancements:
    • Consistent colour per agent (already in agent.color — use in embeds)
    • Agent status indicator: 🟢 online, 🟡 busy, 🔴 error
    • Brief personality tag in responses: "[Security perspective]" for Kane,
      "[Performance angle]" for Kai — helps users know which lens is applied


  8.10 UPGRADES CHANNEL — Better discovery and triage UX
  ─────────────────────────────────────────────────────
  The 6-hour auto-triage is good. Improvements:
    • Each upgrade post should be an embed with:
        Title: upgrade name
        Priority: 🔴 high / 🟡 medium / 🟢 low
        Agent: who proposed it
        Status: pending / accepted / deferred
    • Add reaction voting: 👍 = support, 👎 = oppose
    • Show progress when an upgrade is being implemented:
        "🔧 Upgrade #47 is being implemented by Ace — ETA: next deploy"
