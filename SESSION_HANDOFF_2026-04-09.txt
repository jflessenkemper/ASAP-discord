## ISSUES FOUND & WHAT NEEDS FIXING

### Issue 1: ace-only-delegation test FAILS (expectNone pattern too strict)
The test expects Riley NOT to mention any specialist directly:
  expectNone: [/@kane|@max|@raj|@elena|@kai|@jude|@liv|@harper|@mia|@leo/i]
But Riley's response likely mentions Ace AND mentions specialists by name (not @mention).
The expectAll pattern requires [/@ace|developer/i] which may also be too loose.
FIX NEEDED: This is an existing test (not new), but it times out because the bot's reply
doesn't match the expected shape. Investigate what Riley actually replied — may need to
relax the expectNone or make the prompt clearer about not @mentioning specialists.

### Issue 2: goal-tracking test FAILS (timeout)
Riley doesn't produce output matching: /goal|fuel|status|set|tracking|review/i
This could be because:
- Riley's goal-tracking behavior requires actual ACTION tags that aren't echoed in text
- The bot may be producing responses that get cut off (Discord 2000 char limit)
- Riley might delegate to Ace instead of answering directly
FIX NEEDED: Check Discord channel to see what Riley actually replied. May need to broaden
expectAny patterns or adjust the prompt to be more direct.

### Issue 3: memory-write test FAILS (timeout)
Prompt: "Write a note to repo memory: 'smoke test validated all agents' using memory_write"
expectToolAudit: ['memory_write']
The bot likely doesn't have a tool literally called `memory_write` — the actual tool names
might be `repo_memory_index`, `repo_memory_search`, or something else.
FIX NEEDED: Check the actual tool names available. The tool may be called something like
`write_file` targeting a memory path, or `repo_memory_write`. Verify in the tools.ts file
and update the test prompt and expectToolAudit accordingly.

### Issue 4: Many tests need retries
6 out of 14 completed tests needed retries, suggesting:
- 600ms poll interval might be too aggressive (causing missed responses)
- 180s timeout is fine for most but tight for orchestration (multi-agent chains)
- The bot takes 7-16s for simple Riley tasks but 50-90s for orchestration
CONSIDER: Increasing orchestration timeout to 300s, or adding category-specific timeouts.

### Issue 5: Messages getting cut off (Discord 2000 char limit)
Several responses end with "…" indicating truncation. The test's extractReplyText()
already handles this (slices to 2000 chars), but validation patterns may miss content
that was cut off in the actual Discord message.
CONSIDER: For long responses, the bot may split across multiple messages. The test
only checks content of individual messages, not concatenated multi-message responses.

### Issue 6: Phase 2 parallel execution untested
The parallel agent-channel execution (Phase 2) hadn't started yet. Key concerns:
- Will all 12 agent channels receive messages simultaneously without rate limiting?
- Will the independent per-channel queues handle concurrent load?
- Discord API rate limits may throttle parallel message sending
WATCH FOR: Rate limit errors (429s) in Phase 2 output.
