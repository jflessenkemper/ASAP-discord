Architecture review findings from this session
1. Self-improvement is still in-band with the main user request path.
   - In groupchat flow, Opus completion immediately triggers loop adapters and Operations Manager stewardship instead of enqueueing background work.
   - Risk: user-facing latency, more timeout exposure, and coupling between execution and maintenance.
2. Groupchat orchestration is globally single-threaded and globally cancellable.
   - One process-wide queue plus one active abort controller means one active run can block or cancel others.
   - Acceptable for single-operator mode, but brittle for shared or more autonomous usage.
3. Ownership boundary around self-improvement is still blurred in code.
   - Docs/diagrams show it under the agent manager, but Opus still builds the self-improvement packet.
   - Suggested direction: Opus emits execution outcomes; agent manager derives stewardship packets.
4. Loop architecture is ahead of implementation reality.
   - Docs present nine loops feeding the engine uniformly.
   - loopAdapters.ts still marks some loops as event-driven/non-callable, so not all loops are true callable adapters yet.

Recommended architecture follow-up work
1. Move self-improvement off the user request critical path.
  - Have Opus emit an execution outcome event or durable outbox record.
  - Run loop adapters and Operations Manager stewardship from a background worker with retries.
2. Tighten ownership boundaries.
  - Opus should emit execution outcomes, evidence, and completion status only.
  - Riley the agent manager should derive self-improvement packets and stewardship work from those outcomes.
3. Replace the global groupchat queue model if broader concurrency is desired.
  - Shift from one process-wide messageQueue/abort controller to per-thread or per-goal mailboxes persisted in Postgres.
  - Make steps idempotent so retries and restarts do not duplicate side effects.
4. Make loop capabilities explicit.
  - Split loops into callable loops versus observed/event-driven loops in docs and code.
  - Either add callable adapters for the loops shown as engine inputs, or stop presenting all nine as equivalent callable remediation paths.
5. If resuming architecture design work, start with an updated diagram.
  - Draw a decoupled agent-manager and background stewardship worker architecture before changing runtime code.
